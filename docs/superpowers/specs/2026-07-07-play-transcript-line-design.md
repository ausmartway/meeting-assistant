# Play a transcript line's audio (speaker verification)

**Date:** 2026-07-07
**Status:** Approved
**Branch:** `feature/play-transcript-line`

## Problem

Speaker naming is best-effort (OCR + voice matching), and the user needs a fast
way to check "is this really Dinesh?" The transcript shows *what* was said but
offers no way to hear *who* said it. Clicking a line and hearing that moment of
audio makes verifying / correcting names (R8/R9) practical.

Constraint discovered in design: only `transcript.md` persists today. The exact
segment boundaries and — critically — **which file** a line came from are
discarded after processing. A named speaker can be remote (`system.wav`) or
in-room (`mic.wav`), so the label alone cannot locate the audio.

## Decisions (made with the owner)

- **Persist segments going forward; best-effort fallback for old recordings.**
- **Hover play button per line** (one click to hear, click again to stop).

## Design

### 1. Segment persistence

- `LabeledSegment` gains `channel: AudioChannel?` — backward-compatible Codable
  (decode-if-present), carried through `SpeakerFuser.fuse` from the source
  `TranscriptSegment`s.
- After saving `transcript.md`, `MeetingProcessor` also writes **`segments.json`**
  (the ordered `[LabeledSegment]`) into the meeting bundle via
  `MeetingStore.saveSegments(_:for:)` / `segments(for:)`.
- Lifecycle: deleted with the bundle (R20); kept when retention expires the WAVs
  (harmless, few KB); regenerated by "Transcript Again".

### 2. Locating a line's audio — `TranscriptAudioLocator` (pure, MeetingKit)

Input: parsed `TranscriptParser.Turn`s, optional `[LabeledSegment]`,
`recordedAt`, the local user's display name, and the bundle's audio file names.
Output per turn: `ClipLocation { fileName: String, start: TimeInterval,
end: TimeInterval }?`.

- **Precise path** (segments present): the formatter merges consecutive
  same-speaker segments into one rendered line, so the locator groups segments
  the same way (real meetings always have same-speaker adjacency) — turn *i*
  pairs with group *i*, whose clip spans the group's first segment's start to
  its last segment's end. Sanity-check speaker equality and the rendered
  timestamp of the group's first segment before trusting it. Any mismatch →
  fallback path.
- **Fallback path** (no segments / mismatch): offset = turn's `HH:mm:ss`
  wall-clock stamp minus `recordedAt`'s clock time, +24 h if negative (midnight
  wrap). End = next turn's offset (capped at 30 s), or offset + 15 s for the
  last turn. Channel guess: speaker == local user name → `mic.wav`, else
  `system.wav`. Documented as best-effort: in-room named speakers can map to
  the wrong file until the meeting is re-transcribed.
- Segments with a nil channel (legacy segments.json will not exist, but a nil
  from any future producer) use the same label-based channel guess.

### 3. Playback — `ClipPlayer` (app target)

`ObservableObject` wrapping `AVAudioPlayer`:
- `play(url:from:to:turnID:)` — seeks to `from`, plays, auto-stops at `to`
  (timer/deadline); publishes `playingTurnID: String?`.
- Starting a new clip stops the current one; `stop()` stops explicitly.
- Errors (file unreadable) surface as a stopped state, no banner — the feature
  is auxiliary.

### 4. UI — `TranscriptReadingView`

- Hovering a turn reveals a small play/stop button beside the timestamp; click
  plays that turn's clip, click again stops. The playing turn's row is
  highlighted and its button shows stop while active.
- The control is absent entirely when `hasAudio == false` (retention expired
  the WAVs) — same degradation pattern as "Transcript Again".
- The view needs the recording (bundle URLs via `MeetingStore`), the located
  clips, and a shared `ClipPlayer` from the detail view.

## Error handling

- Missing/unparsable `segments.json` → fallback path (never a hard failure).
- Locator returns nil for a turn (unparsable timestamp) → no button on that row.
- Audio file missing at play time → player no-ops to stopped state.

## Testing

Pure + TDD (swift-testing): `TranscriptAudioLocator` (index pairing, sanity
check falling back on mismatch, offset math incl. midnight wrap, end capping,
channel guess, last-turn window), `LabeledSegment` channel round-trip + legacy
decode, `MeetingStore.saveSegments`/`segments(for:)` round-trip. `ClipPlayer`
and hover UI are AVFoundation/SwiftUI — verified by running the app.

## Requirements impact

- New **R27 — Verify a speaker by ear.** Clicking play on a transcript line
  plays that line's audio: precise for meetings processed after this feature,
  best-effort (timestamp-derived, channel-guessed) for older ones;
  re-transcribing upgrades a meeting to precise. Unavailable once retention has
  expired the audio, matching R20/R22 behavior.
- R20's "delete removes everything" implicitly covers `segments.json` (bundle
  directory delete). No change needed.
